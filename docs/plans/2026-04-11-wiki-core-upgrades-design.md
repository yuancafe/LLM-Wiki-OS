## Problem Statement

llm-wiki-skill 已有完整的知识管理工作流（init/ingest/batch-ingest/query/digest/lint/status/graph），但在以下方面存在明显短板：

1. **缺乏方向性上下文** — wiki 没有"目标"声明，LLM 在 ingest/query 时缺少研究方向指引
2. **单步 ingest 质量瓶颈** — 一次 LLM 调用同时做分析和生成，互相干扰
3. **无去重机制** — batch-ingest 重复处理已消化的素材，浪费 token
4. **缺少删除工作流** — 素材只能增不能删，没有数据生命周期管理

竞品 nashsu/llm_wiki（Tauri 桌面应用）在这四个方向上都有成熟方案，可以借鉴其核心思路适配到 skill 架构。

## What Makes This Cool

这些改动让 wiki 从"被动堆砌"变成"有方向的、可维护的知识系统"。purpose.md 给每次操作加了指南针，两步 ingest 让分析质量飞跃，缓存消除浪费，删除让知识库可以呼吸。四项改动叠加，系统的专业度会明显上台阶。

## Constraints

- 所有改动在 SKILL.md + shell 脚本层面，不引入新运行时框架
- 不影响已有 wiki 的向后兼容（新功能是可选的附加能力）
- SHA256 缓存存放在 wiki 本地（`.wiki-cache.json`），不用外部数据库
- 级联删除作为新增第 9 个工作流（delete）加入 SKILL.md

## Premises

1. **SKILL.md 是可执行规格** — 改 SKILL.md 就是改产品，改脚本只是改基础设施。用户已同意。
2. **purpose.md 纯新增** — 旧 wiki 没有 purpose.md 不影响任何功能，新 wiki init 时生成模板。用户已同意。
3. **两步 ingest 改流程描述** — 核心是 SKILL.md 里 ingest workflow 的重构，脚本不变。用户已同意。
4. **SHA256 缓存本地存储** — `.wiki-cache.json` 放在 wiki 根目录，结构简单可读。用户已同意。
5. **delete 是独立工作流** — 不和现有 workflow 耦合，用户主动触发。用户已同意。

## Approaches Considered

### Approach A: 四项一起做（推荐）

一次性实现全部四项改进，因为它们互相关联：
- init 改造同时加 purpose.md 和缓存初始化
- ingest 重构天然集成两步流程 + 缓存读写
- delete 补齐生命周期

**Effort:** M
**Risk:** Medium（ingest 是核心工作流，同时被改进 2 和改进 3 修改，需要仔细编排改动顺序）
**Pros:**
- 一次改完 SKILL.md，避免多次重构同一区域
- 四项改动互相验证，设计一致性好
- 测试可以一起写

**Cons:**
- 单次 diff 较大，需要仔细 review
- ingest 工作流同时被两步流程和缓存检查修改，交互效果需要验证

### Approach B: P0+P1 先行，P2+P3 后续

先做 purpose.md 和两步 ingest（最高价值的两项），缓存和删除后补。

**Effort:** S (first batch)
**Risk:** Low
**Pros:**
- 最快拿到核心价值
- 两步 ingest 是最大的单点提升

**Cons:**
- batch-ingest 重复浪费问题继续存在
- 数据生命周期不完整

### Approach C: 每项独立提交

四项改动各自独立 commit，每次只改一个方面。

**Effort:** M
**Risk:** Low
**Pros:**
- 每次改动最小化，review 最容易
- 出问题回滚范围小

**Cons:**
- SKILL.md 会被反复编辑，每次都有 merge 风险
- init/ingest 相关的改动分散在多个 commit，整体性差

**RECOMMENDATION:** Approach A。四项改动都在同一个文件（SKILL.md）的核心区域（init、ingest、新增 delete），拆开做反而增加重复编辑的成本。

## 改进 1: purpose.md — 方向性上下文

### 改动内容

**1. 新增模板文件** `templates/purpose-template.md`（中文版）和 `templates/purpose-en-template.md`（英文版）：

```markdown
# 研究目的与方向

## 核心目标
<!-- 这个知识库要解决什么问题？帮助谁？ -->
[待填写]

## 关键问题
<!-- 你最想知道答案的 3-5 个问题 -->
1. [待填写]
2. [待填写]
3. [待填写]

## 研究范围
<!-- 涵盖哪些领域/主题？不涵盖什么？ -->
**涵盖：** [待填写]
**不涵盖：** [待填写]
```

**2. SKILL.md init workflow 修改：**
- 在创建目录结构后，额外生成 `purpose.md`
- 提示用户填写核心目标和关键问题
- 和 `.wiki-schema.md` 同级存放

**3. SKILL.md ingest workflow 修改（首期范围）：**
- 在"读取上下文"步骤中，明确加入读取 `purpose.md`
- 优先级：`purpose.md` > `.wiki-schema.md` > `index.md`
- ingest 时用 purpose.md 指导实体/主题的分类和权重
- **query 和 digest 的 purpose.md 集成作为后续增强**，等用户实际填写了 purpose.md 后再验证价值（YAGNI）

**4. wiki-compat.sh 修改：**
- 旧 wiki 没有 purpose.md 时，不影响任何功能（向后兼容）
- status workflow 报告 purpose.md 是否存在

### 为什么有效

现在每次 ingest，LLM 只知道"主题是什么"（来自 `.wiki-schema.md`），不知道"用户为什么要建这个 wiki"。purpose.md 补充了"为什么"和"往哪走"，让 LLM 的分类、取舍、摘要更有针对性。

## 改进 2: 两步链式思考 Ingest

### 当前流程（单步）

```
URL/文件 → 提取内容 → [一次 LLM 调用] → 生成全部 wiki 页面
```

一次调用里同时要求 LLM 做分析（提取实体、识别概念、发现矛盾）和生成（写页面、更新 index），两个任务互相争抢注意力。

### 新流程（两步）

```
URL/文件 → 提取内容 → [Step 1: 分析] → [Step 2: 生成] → 写入文件
```

**Step 1 — 结构化分析：**
输入：原始内容 + purpose.md + 现有 wiki 结构（index.md 概要）
输出：JSON 格式的分析结果
```json
{
  "source_summary": "一句话概括",
  "entities": [{"name": "xxx", "type": "person|concept|tool|org", "relevance": "high|medium|low"}],
  "topics": [{"name": "xxx", "importance": "high|medium|low"}],
  "connections": [{"from": "xxx", "to": "xxx", "type": "对比|因果|关联"}],
  "contradictions": [{"claim_a": "...", "claim_b": "...", "context": "..."}],
  "new_vs_existing": {"new_entities": [...], "updates": [...]}
}
```

**Step 2 — 页面生成：**
输入：原始内容 + purpose.md + Step 1 的分析结果 + 现有相关 wiki 页面
输出：所有需要创建/更新的 wiki 页面内容

### SKILL.md 改动

ingest workflow 拆分为两步：
- **长内容（>1000 字符）**：走两步流程，Step 1 分析 + Step 2 生成。Step 1 替代原流程的"阅读素材，提取关键信息"步骤，Step 2 替代原流程的"生成素材摘要页"到"更新 log"步骤。
- **短内容（<=1000 字符）**：保持简化流程不变（不值得两步的 token 开销）
- Step 1 的分析结果不持久化（用完即丢），只在同一个 ingest 流程内传递
- **容错设计：** 如果 Step 1 的输出不是有效 JSON，或缺少必需字段（entities、topics），自动回退到单步流程。不因为格式问题阻塞整个 ingest

### 为什么有效

分析任务需要"读和思考"，生成任务需要"写和组织"。拆开后，Step 1 专注理解，Step 2 拿着结构化的分析结果专注表达。这和人类写综述的流程一致：先通读做笔记，再动笔写文章。

## 改进 3: SHA256 增量缓存

### 设计

**缓存文件：** `.wiki-cache.json`，存放在 wiki 根目录

```json
{
  "version": 1,
  "entries": {
    "raw/articles/2024-01-15-ai-article.md": {
      "hash": "sha256:abc123...",
      "ingested_at": "2024-01-15T10:30:00Z",
      "source_page": "wiki/sources/2024-01-15-ai-article.md"
    }
  }
}
```

**Key 计算：** `sha256(文件相对路径 + 文件内容)` — 路径+内容一起哈希，避免同名不同内容的误判。

### 脚本改动

新增 `scripts/cache.sh`，提供三个子命令：
- `check <file>` — 返回 `HIT`（已缓存且 source_page 存在）或 `MISS`（需要处理）。HIT 时额外验证 source_page 文件是否仍在磁盘上（可能被级联删除移走），不在则降级为 MISS
- `update <file> <source_page>` — 处理完成后写入缓存。hash 由 `cache.sh update` 内部通过 `sha256(相对路径 + 文件内容)` 自动计算，调用方只需传 `<file>` 和 `<source_page>`
- `invalidate <file>` — 删除某条目的缓存（用于强制重新处理或级联删除后清理）

### SKILL.md 改动

- **ingest workflow：** 在"提取内容"之后、"LLM 处理"之前，加入缓存检查。HIT 则跳过 LLM 调用，直接读已有 wiki 页面。MISS 则正常走两步流程，完成后 update 缓存。缓存 HIT 的语义是乐观锁：假设 wiki 状态与上次处理一致。如果中间有过手动编辑 wiki 页面或级联删除，缓存可能过时。`cache.sh check` 会验证 source_page 文件是否仍存在，不存在则降级为 MISS。
- **batch-ingest workflow：** 每个文件先 `cache check`，只处理 MISS 的文件。处理完成后报告统计：`已跳过 N 个（无变化），处理 M 个（新增/更新）`。
- **delete workflow：** 删除素材时 `cache invalidate` 对应条目。
- **init workflow：** 创建 wiki 时初始化空的 `.wiki-cache.json`。

### 向后兼容

- 旧 wiki 没有 `.wiki-cache.json` 时，所有文件都是 MISS（等效于没有缓存）
- wiki-compat.sh 检测并报告缓存状态

### 为什么有效

batch-ingest 10 个文件，其中 7 个没变过。没有缓存就要白白花 7 次完整 LLM 调用的 token。有了缓存，只处理真正变化的 3 个。

## 改进 4: 级联删除（Delete Workflow）

### 设计

新增第 9 个工作流：`delete`。

**触发词：** "delete source", "remove", "删除素材", "移除"

### 流程

```
用户指定素材 → 确认范围 → 级联清理 → 更新缓存
```

**Step 1 — 识别目标：**
用户说"删除那个 AI 文章"，LLM 在 `raw/` 中搜索匹配的文件，展示找到的文件让用户确认。

**Step 2 — 分析影响范围（脚本搜索 + LLM 决策）：**
引用扫描由脚本完成（不用 LLM，保证全量精确），决策由 LLM 完成。

脚本搜索逻辑（新增 `scripts/delete-helper.sh` 或集成到 `cache.sh`）：
```bash
# 搜索所有包含该素材文件名的 wiki 页面
grep -rl "素材文件名" wiki/ --include="*.md"
# 搜索 sources: frontmatter 中包含该素材的页面
grep -rl "sources:.*素材文件名" wiki/ --include="*.md"
# 搜索 [source: ...](...) 链接指向该素材的页面
grep -rl "\[source:.*\](.*素材文件名.*)" wiki/ --include="*.md"
```

LLM 拿到脚本返回的文件列表后，逐页判断需要删除还是修改（保留页面只删引用）。

**Step 3 — 级联清理：**

| 清理项 | 规则 |
|--------|------|
| raw/ 原始文件 | 删除 |
| wiki/sources/ 对应页面 | 删除 |
| wiki/entities/ 引用该素材的实体 | 保留页面，删除该素材的引用段落（通过 frontmatter `sources:` 或行内 `[source:](...)` 链接定位。如果无法确定来源归属，保留内容）。如果该实体只被这一个素材引用，提示用户是否也删除实体页 |
| wiki/topics/ 引用该素材的主题 | 同 entity 规则 |
| wiki/comparisons/ 引用该素材的对比 | 同 entity 规则 |
| wiki/synthesis/ 引用该素材的综合 | 同 entity 规则 |
| wiki/index.md | 删除该素材相关的条目 |
| wiki/log.md | 添加一条删除操作日志 |
| wiki/overview.md | 标记为需要重新生成 |
| [[wikilinks]] 断链 | 用 grep 脚本扫描所有页面，清理指向已删除页面的链接 |
| .wiki-cache.json | 删除对应条目 |

**Step 4 — 报告：**
```
已删除：
  - raw/articles/2024-01-15-ai-article.md
  - wiki/sources/2024-01-15-ai-article.md
已更新（移除引用）：
  - wiki/entities/AI-Agent.md
  - wiki/topics/大语言模型.md
需要重新生成：
  - wiki/overview.md
```

### 安全机制

- **确认步骤：** 影响超过 5 个页面时，列出所有受影响页面让用户二次确认
- **不自动删除独立实体：** 如果一个实体被多个素材引用，只删引用不删页面
- **可逆提示：** 告知用户"如果需要恢复，可以从 raw/ 重新 ingest"

### 工作流路由表更新

在 SKILL.md 的工作流路由表中增加一行：
```
| '删除素材'、'remove'、'delete source'、'移除' | -> **delete** |
```

## 改进 5: 置信度标注（Confidence Labeling）

### 来源

借鉴自 graphify 的置信度标注系统。graphify 给每个图谱关系标注 EXTRACTED/INFERRED/AMBIGUOUS，让用户知道哪些是事实、哪些是猜测。

### 设计

在两步 ingest 的 Step 1 分析结果中，为每个实体关系和连接增加置信度标签：

- **EXTRACTED** — 直接从源文件提取的事实（如"张三是公司CEO"出现在原文中）
- **INFERRED** — LLM 基于上下文推断的关系（如根据多篇文章推断两个概念有关联）
- **AMBIGUOUS** — LLM 不确定的关系，需要后续验证
- **UNVERIFIED** — 降级模式专用（见容错设计）

### SKILL.md 改动

在两步 ingest 的 Step 1 JSON 输出格式中，entities 和 connections 增加 confidence 字段。**confidence 列为必需字段**，与 entities、topics 同级，缺失则触发回退：
```json
{
  "entities": [{"name": "xxx", "type": "concept", "relevance": "high", "confidence": "EXTRACTED"}],
  "connections": [{"from": "A", "to": "B", "type": "因果", "confidence": "INFERRED"}]
}
```

生成的 wiki 页面中，INFERRED 和 AMBIGUOUS 的关系用 HTML 注释标注：
```markdown
<!-- confidence: INFERRED -->
A 和 B 之间存在因果关系...
<!-- confidence: AMBIGUOUS -->
这两者可能有关联，待验证...
```

**降级场景处理：** 当 Step 1 JSON 格式异常触发单步回退时，所有生成内容统一标注为 `<!-- confidence: UNVERIFIED -->`，并在页面顶部加注释说明本次处理因格式问题降级。不出现"部分有标注、部分没有"的不一致状态。

lint workflow 增加置信度报告：统计各类置信度的数量，高亮 AMBIGUOUS 条目供用户验证。增加审计逻辑：抽查标注为 EXTRACTED 的条目是否能在原始素材中找到原文，防止 LLM 高估确定性。

### 为什么有效

没有置信度标注的 wiki，用户分不清哪些是原文提取的事实、哪些是 AI 的猜测。标注后，用户可以有针对性地验证不确定的内容，提升知识库的整体可信度。

## 改进 6: Query 结果持久化（记忆反馈循环）

### 来源

借鉴自 graphify 的记忆反馈循环。graphify 把 Q&A 结果存回知识库，下次处理时并入。

### 设计

在 query workflow 末尾增加一步：将综合回答保存为 `wiki/queries/{date}-{topic}.md`。

页面格式（注意 `derived: true` 标记）：
```markdown
---
type: query
derived: true
title: "关于 XXX 的综合回答"
created: 2024-01-15
updated: 2024-01-15
tags: [topic1, topic2]
sources: ["source1.md", "source2.md"]
related: ["entity1", "entity2"]
---

## 问题
用户提出的原始问题

## 回答
LLM 生成的综合回答

## 引用来源
- [source: ...](../sources/xxx.md)
```

### 自引用防护（关键）

query 页面是 LLM 综合的二级产物，不是一手素材。为防止自引用膨胀循环：

1. **二级来源标记：** frontmatter 中 `derived: true` 明确标识这是衍生内容
2. **ingest 降权处理：** 两步 ingest 的 Step 1 分析时，query 页面视为二级来源，relevance 自动降权。不作为主要知识来源
3. **置信度联动：** 源自 query 页面的信息在新生成的页面中统一标注为 `INFERRED`，即使 query 页面原文标注为 EXTRACTED
4. **不自动纳入：** ingest 流程不主动扫描 `wiki/queries/` 目录。只有 LLM 在分析现有 wiki 结构时主动判断需要才读取

### 重复检测

query 持久化前，先在 `wiki/queries/` 下搜索是否已有同主题页面（通过 frontmatter tags 和 title 匹配）：
- 如果已有，提示用户是否更新现有页面（替代旧版本，旧版移至 `superseded-by` 标记）还是创建新页面
- 文件名用 `{date}-{short-hash}` 而非纯 topic，避免命名冲突

### SKILL.md 改动

- query workflow 末尾：如果回答引用了 3 个以上来源的综合分析，提示用户"是否保存此回答到知识库？"（用具体数量规则而非模糊的"价值较高"判断）
- 用户确认后，执行重复检测，按模板生成 query 页面
- 更新 index.md 加入新条目

### 为什么有效

常见问题的回答越积越多，知识库从"素材仓库"变成"活的知识系统"。二级来源标记和降权处理防止了 LLM 把自己的综合结果当一手事实引用的膨胀问题。

## 改进 7: 自动注入 Wiki 上下文

### 来源

借鉴自 graphify 的 hook 模式和 claude-mem 插件的 SessionStart 模式。

### Phase 0 验证结果

原设计使用 PreToolUse hook + `Grep|Glob|Read` 匹配器 + echo 纯文本输出。经验证：

1. **echo 纯文本不会被 LLM 看到** — hook 输出必须是结构化 JSON（`{"systemMessage": "..."}` 或 `{"hookSpecificOutput": {"additionalContext": "..."}}`），纯文本等于对着空气说话
2. **PreToolUse 每次工具调用都触发** — 在典型会话中触发几十次，每次注入提醒是噪音轰炸
3. **结论：PreToolUse 不适合这个场景**

### 新方案：SessionStart hook

改用 **SessionStart hook**。每次会话开始触发一次，通过 `hookSpecificOutput.additionalContext` 注入 wiki 路径信息。

### 平台支持现状

| 平台 | Hook 支持 | 方案 |
|------|----------|------|
| Claude Code | SessionStart hook（已验证可行） | 注册 hook，调用脚本输出 JSON |
| Codex | 不支持 SessionStart hook | 在 AGENTS.md 开头注入 wiki 上下文提示 |
| OpenClaw | 内部 hook 机制（不兼容） | 暂不支持 |

### 设计

**Claude Code 平台：**

修改 `~/.claude/settings.json`，注册 SessionStart hook：

```json
{
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "bash ~/.claude/skills/llm-wiki/scripts/hook-session-start.sh"
      }]
    }]
  }
}
```

新增 `scripts/hook-session-start.sh`：
```bash
#!/usr/bin/env bash
# SessionStart hook: 会话开始时注入 wiki 上下文（只触发一次）
WIKI_PATH=""
[ -f ~/.llm-wiki-path ] && WIKI_PATH=$(cat ~/.llm-wiki-path)

# 尝试当前目录
if [ -z "$WIKI_PATH" ] && [ -f .wiki-schema.md ]; then
  WIKI_PATH="$(pwd)"
fi

# 没有 wiki，静默返回空 JSON
[ -z "$WIKI_PATH" ] && echo '{}' && exit 0
[ ! -f "$WIKI_PATH/.wiki-schema.md" ] && echo '{}' && exit 0

# 通过 additionalContext 注入（LLM 会看到这段文字）
printf '{"hookSpecificOutput":{"hookEventName":"SessionStart","additionalContext":"[llm-wiki] 检测到知识库: %s/index.md，回答问题时优先查阅 wiki 内容获取上下文"}}\n' "$WIKI_PATH"
```

**Codex 平台：**
在安装 skill 时，在项目的 `AGENTS.md` 开头追加一行：
```
<!-- llm-wiki context: 如有知识库，优先查阅 wiki/index.md -->
```

### settings.json 安全机制

- **备份：** 修改前复制到 `settings.json.bak.llm-wiki`
- **幂等：** 用 `jq` 检查 hook 是否已存在（按 command 字符串匹配），已存在则跳过
- **合并：** 用 `jq` 做数组追加而非覆盖，保留用户已有的其他 hooks
- **回滚：** 提供 `--uninstall-hooks` 标志，从 settings.json 中移除 llm-wiki 的 hook 条目

### install.sh 行为

- `install.sh --platform claude` — 默认不注册 hook
- `install.sh --platform claude --install-hooks` — 安装 skill 后注册 SessionStart hook
- `install.sh --install-hooks`（skill 已安装）— 只注册 hook，不重新安装 skill 文件
- `install.sh --uninstall-hooks` — 移除 hook（从 settings.json 中删除 llm-wiki 条目）

### 为什么有效

现在用户必须主动说"查一下 wiki 里的 X"才能触发知识库。SessionStart hook 每次会话开始提醒一次，不吵不闹，AI agent 自然会在需要时查阅 wiki。比 PreToolUse 安静几十倍。
